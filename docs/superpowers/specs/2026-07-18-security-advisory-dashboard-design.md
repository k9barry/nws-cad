# Consolidated security-advisory dashboard (programs & libraries)

**Date:** 2026-07-18
**Status:** Draft (design) — for review

## Problem

The system has **no runtime visibility into the security posture of the software it runs on.** `.github/workflows/security.yml` is a stub (every job just `echo`s), there is no `composer audit` anywhere in CI, no `dependabot.yml`, and the vendored frontend libraries under `public/assets/vendor/` have no manifest at all. If a CVE lands against Monolog, MySQL, Leaflet, or a Debian package baked into the image, nobody finds out from this application.

At the same time the stack is deployed in two places — on-premise at the **PSAP** and in the **cloud** — running largely the same components, occasionally at different versions or with different databases (MySQL on-prem vs Postgres in cloud, for example). An operator has nowhere to see, in one place, "what am I running, where, and what needs patching first."

## Goal

A single **super-admin dashboard page** that consolidates every program and library used across both the PSAP and cloud environments, checks each against reputable security-advisory sources, and renders the results **sorted by severity** and **grouped/filterable by application (`psap | cloud`)**. Advisory data is refreshed on a **schedule into a cache**; the page renders the cached results instantly and does not depend on outbound network access at view time.

## Non-goals

- **No live polling of remote PSAP nodes.** (Explicitly dropped from the earlier design.) There is no fleet, registry, per-node auth, or inbound reachability requirement. `psap` and `cloud` are *labels on the inventory this repo already describes*, not remote hosts we contact.
- **No auto-remediation.** The page reports; it does not upgrade dependencies or apply patches.
- **No mutation of the ingestion, notifications, or call pipeline.** This is an additive, read-only admin surface.
- **No new distributed architecture.** One page, one manifest, one scheduled scan.

## Terminology: what `psap` and `cloud` mean here

This repository is a **single codebase and a single Docker stack** (`app` watcher + `api` + a database + `dozzle` + `autoheal`). It is deployed to two *environments*:

| Label | Meaning |
|---|---|
| `psap` | The on-premise deployment at the Public Safety Answering Point — the box watching CAD XML on the CIFS `watchfolder`, typically MySQL, often on a locked-down network with restricted egress. |
| `cloud` | The hosted deployment — same application image, possibly a newer release, possibly Postgres. |

Because both run the same image, **most components appear in both environments**; some differ (database engine, optional services). So each inventory component carries an `environments` list (`["psap"]`, `["cloud"]`, or both). An advisory against a shared component surfaces once, tagged with every environment it affects.

## Inventory model

The inventory is **derived where possible, declared where necessary**, and stored as a single canonical file the scan and page both read.

### Sources (auto-derivable — the source of truth stays the lockfiles)

| Component class | Derived from | Ecosystem for lookup |
|---|---|---|
| PHP libraries (runtime) | `composer.lock` (`packages`, not `packages-dev`) | `Packagist` |
| Vendored JS libraries | `public/assets/vendor/VENDORED.md` | `npm` |
| OS packages in the image | `Dockerfile` `apt-get install` list | `Debian` |
| Container images / programs | `docker-compose.yml` `image:` + `Dockerfile` `FROM` | image tag + CVE/NVD |
| PHP runtime & extensions | `Dockerfile` (`php:8.3-cli`, `docker-php-ext-install …`) | version + CVE/NVD |

### Declared supplement — `config/inventory.json`

A checked-in manifest that (a) assigns each component to environments and (b) adds anything not in a lockfile (the DB engine actually run per environment, the cloud release if it differs). Shape:

```json
{
  "components": [
    { "name": "monolog/monolog", "ecosystem": "Packagist", "version": "3.10.0",
      "type": "library", "environments": ["psap", "cloud"], "derived": true },
    { "name": "mysql", "ecosystem": "NVD", "version": "8.0",
      "type": "program", "environments": ["psap"], "derived": false },
    { "name": "postgres", "ecosystem": "NVD", "version": "16",
      "type": "program", "environments": ["cloud"], "derived": false }
  ]
}
```

`derived: true` rows are regenerated by `bin/build-inventory.php` from the lockfiles and must **not** be hand-edited; `derived: false` rows are operator-declared and preserved across regeneration. The build script merges the two and fails loudly if a derived component lost its `environments` assignment (so new deps can't silently escape the scan).

## Advisory sources (the "reputable sources" requirement)

**Primary: [OSV.dev](https://osv.dev)** — Google's Open Source Vulnerabilities database. One free, no-auth batch endpoint (`POST https://api.osv.dev/v1/querybatch`) that federates GitHub Advisory Database (GHSA), NVD/CVE, and the PHP FriendsOfPHP/Packagist advisory DB, and covers the Packagist / npm / Debian ecosystems we need. Returns advisory IDs and CVSS severity — exactly the sort key.

**Secondary (PHP, native): `composer audit --format=json`** — run in CI against `composer.lock`; authoritative for the PHP side and a cross-check on OSV.

**Programs / container images** (MySQL, Postgres, PHP itself, Dozzle) aren't library packages, so they're matched at the product level against **NVD/CVE by version**. For these we additionally surface an **"update-status" signal** — "N releases behind latest" from each image's registry tag list / GitHub Releases — clearly distinguished in the UI from a vulnerability (see severity note below).

Rationale for OSV as the backbone: single reputable aggregator, free, no key to manage, no per-source parsing, covers every ecosystem in our inventory. It replaces what the stubbed `security.yml` was always meant to do.

## Refresh model — scheduled cache (chosen)

The scan runs **on a schedule, off the PSAP box, and writes a cached result file** the page reads. This keeps the locked-down on-prem deployment free of any view-time egress.

**Primary: scheduled GitHub Action** (`.github/workflows/security-advisories.yml`, `schedule: cron` daily + `workflow_dispatch`):

1. `composer install`, then `composer audit --format=json`.
2. Build the inventory (`bin/build-inventory.php`) and POST it to OSV `querybatch`.
3. Normalize + merge into `var/security/advisories.json` and commit it (or publish as a release artifact the deploy pulls). Because inventory is derived from the repo's own lockfiles, CI is the correct and sufficient place to evaluate it — the scan reflects exactly what a given release ships.

**Optional runtime fallback:** a `bin/refresh-advisories.php` invoked from the existing `FileWatcher` tick (guarded by a TTL, default 24h) for deployments that *do* have egress and want fresher data than the release cadence. Reuses the same `OsvClient`. Off by default.

The controller/page **only ever reads the cache file** — never calls OSV inline — so page load is instant and works with zero connectivity.

## Data shape

Each normalized advisory row:

```json
{
  "component": "monolog/monolog",
  "type": "library",
  "environments": ["psap", "cloud"],
  "installed_version": "3.10.0",
  "advisory_id": "GHSA-xxxx-xxxx-xxxx",
  "severity": "HIGH",
  "cvss_score": 7.5,
  "summary": "…",
  "fixed_version": "3.10.1",
  "source_url": "https://osv.dev/vulnerability/GHSA-…",
  "kind": "vulnerability"
}
```

`kind` is `vulnerability` (from OSV/audit/NVD) or `update-available` (the "N behind" signal for images/programs). The cache file also carries `{ "generated_at": <ISO8601>, "sources": [...], "component_count": N }` so the page can show data freshness and warn when stale.

## Severity & sorting

- Severity buckets: `CRITICAL > HIGH > MEDIUM > LOW > UNKNOWN`, derived from CVSS where present.
- **Primary sort: severity desc.** Secondary: environment, then component name.
- `update-available` rows are **not** given a CVSS severity — they sort into a separate `Updates` bucket below vulnerabilities, so "you're a release behind" never masquerades as a CVE. (Called out because the app's own release-currency is update-status, not a vulnerability.)

## Authorization

Dashboard views have **zero auth today** — a super-admin page is exactly where that gap cannot ship. Reuse the established `LogsController::checkAccess()` pattern:

- New env var **`SUPERADMIN_USERS`** (csv) → `security.superadmin_users` config, CSV-parsed like `logs.admin_users`.
- Gate both the `/security` view include (in `public/index.php`) and the `GET /api/security/advisories` controller on `Identity::current()->user ∈ SUPERADMIN_USERS`.
- **Fail-secure in production:** empty allowlist in prod = deny; non-prod stays open (identical to `LogsController`).

## UI

- New nav entry (rendered only when the caller is authorized), route `/security` → `src/Dashboard/Views/security.php`.
- Default view: **table sorted by severity**, columns `Severity | Component | Type | Application | Installed | Fixed | Advisory`, with a severity color scale (reuse Bootstrap contextual classes) and an **Application filter** (`All | psap | cloud`) plus a type filter (`program | library`). Card layout (one card per advisory, grouped by severity) offered as an alternate view toggle.
- A freshness banner: "Last scanned <relative time>" with a warning style when older than the refresh cadence × 2.
- All CAD-independent, but any interpolated value still goes through `Dashboard.safeHtml` per the JS conventions.

## Configuration

| Var | Default | Purpose |
|---|---|---|
| `SUPERADMIN_USERS` | *(empty)* | CSV allowlist of identity-header users permitted to view the security page / API. Empty in prod = deny. |
| `SECURITY_ADVISORIES_ENABLED` | `true` | Master switch for the page + API. |

## Pros / cons & tradeoffs

**Pros**
- Fills a genuine, currently-empty gap (stub `security.yml`, no `composer audit`, no dependency visibility).
- Radically simpler than the dropped fleet design — no remote auth, firewalls, or registry.
- One reputable aggregator (OSV) covers every ecosystem we ship; `composer audit` cross-checks PHP.
- Egress-free at view time; the on-prem box needs no internet to render the page.
- Additive and read-only — cannot affect ingestion, notifications, or calls.

**Cons / risks**
- **Inventory accuracy is the whole game.** Derived rows stay honest automatically; declared rows (DB engine per environment, image versions) drift unless updating them is part of the release habit. A drifted manifest gives *false confidence* — mitigated by auto-deriving everything possible and failing the build when a derived component lacks an environment.
- **Vendored JS versions are currently imprecise** (`10.x`, `4.x`) and unscannable until pinned.
- **Programs/images aren't clean package lookups** — NVD/CVE-by-version matching is coarser and noisier than Packagist/npm; the `update-available` signal helps but isn't a vulnerability verdict.
- **Cache staleness** — data is only as fresh as the last scheduled scan; the freshness banner makes this visible rather than silent.
- Another authorization surface to get right; reusing `checkAccess` keeps it consistent but it must be tested fail-secure.

## Open decisions (need your call before implementation)

1. **Cache delivery:** commit `var/security/advisories.json` into the repo from the Action, or publish it as a release artifact the deploy pulls? (Committing is simplest and works with zero deploy-time egress; artifact keeps the repo clean.)
2. **Programs/images scope:** include OS packages + container images in v1 (noisier, NVD-based), or ship v1 with just the PHP + JS libraries (clean OSV coverage) and add programs in v2?
3. **Runtime refresh:** ship the optional `FileWatcher`-tick refresh now, or CI-only for v1?

## Rollout phases

- **Phase 0 (prerequisite):** pin exact vendored JS versions in `VENDORED.md`; expose the real app version (read `VERSION`, fix the stale `1.0.0` in `public/api.php`).
- **Phase 1:** inventory build + OSV/`composer audit` scan in a scheduled Action → cached JSON; authed `/api/security/advisories` + the page (libraries only). Delivers value with clean coverage.
- **Phase 2:** add programs/images + the `update-available` signal, and the optional runtime refresh.
